# 空项目基础保障模式（Baseline Bootstrap）

当项目接近空目录，且用户只给出一句话需求、还不足以判断明确技术栈时，你不能因为信息不足而不给安装建议。此时必须进入基础保障模式，为项目补一层最小但高价值的 AI 协作能力。

## 触发条件
命中以下多数条件时，进入基础保障模式：
- 根目录接近空目录，只有少量说明文件或隐藏目录
- 不存在明显技术栈信号，例如：`package.json`、`pnpm-workspace.yaml`、`pyproject.toml`、`requirements.txt`、`go.mod`、`Cargo.toml`、`pom.xml`、`build.gradle`、`pubspec.yaml`
- 不存在现成 AI 配置，例如：`CLAUDE.md`、`.cursorrules`、`.windsurfrules`、`agent.md`
- 用户需求只有一句话，且不足以判断垂直栈

## 模式目标
- 给项目补最小可开工协作能力
- 避免因为上下文过少而输出空安装清单
- 保留后续按技术栈继续特化安装的空间

## 基线安装建议原则
安装清单必须满足：
- 高通用
- 低重复
- 低冲突
- 对后续项目演化友好

## 在该模式下至少要做到
1. 生成最小可用的 `CLAUDE.md` 方案
2. 给出一份不为空的基础安装清单
3. 明确告诉用户：这是 baseline，不是最终形态，后续识别到技术栈后还会补专项能力

## 基线清单建议方向
根据当前宿主已有能力与重复项情况，至少从以下类别中选择一组基础能力：
- 通用执行增强
- 计划 / 执行 / 验证闭环能力
- 基础调试或代码质量增强能力

不要在空项目阶段就批量安装大量垂直领域 skill；先保证“能稳定开工”，再等待项目技术栈清晰后继续特化。

## 输出要求
当进入该模式时，你必须显式说明：
- 为什么判断这是空项目 / 一句话需求场景
- 为什么要启用 baseline bootstrap
- 当前推荐的是哪一组基础能力
- 哪些能力要等技术栈明朗后再补
