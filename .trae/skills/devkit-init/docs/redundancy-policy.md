# 冗余检测与禁用

在规划或安装完成后，你必须检查当前已安装能力中的功能重叠情况，并用可解释的判定矩阵说明为什么保留 A、禁用 B，而不是只给结论。

## 必查重叠类别
例如但不限于：
- 多个代码审查 skill
- 多个任务管理 skill
- 多个计划生成 skill
- 多个调试增强 skill
- 多个相同职责的 MCP
- 多个都在做 context 压缩 / 会话治理的工具

## 判定矩阵
至少从以下维度打分或描述：
1. **职责重叠度**：核心能力是否覆盖同一工作流或同一触发场景。
2. **触发重叠度**：是否会在相近指令、相近关键词、相近阶段被同时调用。
3. **输入输出相似度**：是否都接收相似输入，并产出近似类型的结果。
4. **宿主暴露成本**：是否会让 skill 列表、MCP 列表或插件入口明显膨胀。
5. **项目贴合度**：哪个候选项更贴合当前技术栈、项目规模与协作方式。
6. **稳定性与来源可信度**：优先保留官方、高可信、维护更活跃的项。

## 快速判定规则
- 职责重叠度高 + 触发重叠度高：默认视为强冗余，不能同时保留。
- 职责重叠度中等，但一个偏通用、一个偏当前项目专用：可保留项目更贴合的项，另一个标为候补。
- 输入输出相似但触发场景明显不同：可共存，但必须说明边界，避免模型误用。
- 仅名字相似、职责不同：不得误判为冗余，先说明差异再决定。

## 处理原则
- 保留更通用、更稳定、更贴合当前项目的高优先级项。
- 禁用低优先级或重复项。
- 不让多个重复 skill 同时暴露给模型。
- 当两个候选项都可用但边界不清时，优先保留一个主项，另一个仅在用户明确要求时再启用。

## 推荐输出格式
对每组疑似重复项，至少说明：
- 候选 A / 候选 B
- 重叠类别
- 判定矩阵摘要
- 保留项
- 禁用项或候补项
- 原因

## 可选禁用方式
根据实际环境选择最合适方法：
- 在 `settings.json` 中配置禁用
- 重命名或移出对应 `SKILL.md`
- 通过插件/市场配置关闭
- 从 MCP 配置中移除对应 server

## 禁用前风险提示
如果禁用动作会影响多个项目、全局宿主配置或团队共享环境，你必须先说明影响范围，再获得确认。

## 输出要求
必须说明：
- 哪些项重复
- 为什么保留 A 而禁用 B
- 判定依据属于职责、触发、输入输出还是暴露成本冲突
- 禁用动作的影响范围
- 若未立即禁用，后续应如何避免重复暴露

## 接入档位例外
当 `devkit-init` 处于接入档位时，用户已装 skill 视为白名单，不触发冗余告警。这些 skill 标记为 `managed_by: user`，DevKit 不接管其生命周期，也不建议替换或禁用。只有用户显式要求接管时，才纳入冗余判定。
